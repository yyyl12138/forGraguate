## 3. 总体依赖路径

工程依赖顺序：

`契约冻结 -> 本地代码闭环 -> 三节点基础设施 -> 三节点 MySQL -> Fabric 链码真实调用 -> 后端真实 Gateway 与三库联调 -> Filebeat/Vulhub 接入 -> 前端真实演示 -> 篡改与完整性验收`

关键判断：

- 链上只保存批次存证数据，不保存日志明文。
- 三台 MySQL 保存日志明文副本，后端负责同步写入与比对。
- 链码负责批次 Root Hash 存证、查询和 Root 校验。
- 后端是系统编排中心，负责日志解析、三库写入、Merkle Root 计算、Fabric Gateway 调用、三库比对和审计。
- 前端只消费后端 API，不直接访问 Fabric 或 MySQL。
- `MockLedgerGatewayClient` 保留为本地回归与故障隔离手段；真实验收必须通过 Fabric Gateway。

### 5.4.1 阶段 9.5：问题复盘、异步副本改造与漏检点审核

目标：

- 基于阶段 9 暴露的问题，审查并控制真实三库 + Fabric 联调中的系统性风险。
- 将三库写入策略从“串行同步写但没有分布式事务”调整为“`node1` 主写 + 冻结 outbox + `node2/node3` 异步追加副本 + seal 前强门禁”。
- 优先保证进入阶段 10 前，后端真实链路稳定、错误可诊断、文档与代码一致。
- 阶段 9.5 已于 `2026-05-11` 通过；当前可进入阶段 10。

审核范围：

- Maven 依赖树：Fabric Gateway、gRPC、protobuf、Spring Boot 管理版本是否仍有隐性冲突。
- MyBatis 多数据源：mapper 注册、事务管理器、三库写入失败语义、审计记录位置是否清晰。
- SQL 注解：普通注解 SQL 与 `<script>` 动态 SQL 的转义是否还有残留风险。
- 哈希链路：`normalized_message` 的 JSON 存储、canonical 序列化、leaf hash、Merkle Root、测试向量是否一致。
- Fabric Gateway：异常翻译、重复批次、链码参数校验、背书失败 details 是否能正确返回。
- 阶段 9 文档：清理命令、重试策略、链上已有批次说明、env 加载说明是否完整。
- 异步副本同步：`replica_sync_tasks` 冻结 payload、payload hash、HMAC signature、失败重试、stale `IN_PROGRESS` 重置和 node2/node3 离线语义是否稳定。
- seal 门禁：三库未追平、三库记录集合不一致、leaf 不一致、普通明文字段不一致时是否都能拒绝上链。
- 完整性差异分类：`EXTRA_LOG`、`MISSING_LOG`、`MODIFIED_LOG`、`BATCH_ROOT_MISMATCH`、`MULTI_REPLICA_DIVERGENCE` 是否准确，异常节点是否只包含真实异常副本。

已完成改造：

- 新增 `node1.replica_sync_tasks` 作为冻结 outbox，保存同步 payload、payload hash 和 HMAC signature。
- `/api/logs/ingest` 调整为 `node1` 主写并创建 node2/node3 异步同步任务；响应新增副本同步状态。
- 后台同步任务只从冻结 payload 写入 node2/node3，不从 `node1.log_records` 当前状态复制，避免 node1 后续 `DELETE/UPDATE` 被传播到副本。
- `/api/batches/seal` 增加同步 drain 和三库一致性门禁。
- Fabric 重复批次在链上 Root/日志数一致时支持幂等复用 evidence；不一致仍拒绝。
- 降低 Hikari 连接超时，并增加 stale `IN_PROGRESS` 任务重置，改善 node2/node3 离线场景下的反馈速度。
- 引入运行时副本记录验证 digest，覆盖稳定业务字段和明文字段，排除 `source_node` 等物理副本字段；seal 前 digest 不一致则拒绝上链。
- 修复完整性差异分类：seal 后 `node2/node3` 作为可信参考副本，`node1` 额外日志归类为 `EXTRA_LOG`，不会误列可信副本为异常节点。
- 为 Fabric Gateway evaluate/endorse/submit/commit status 增加 deadline，并为 seal 增加关键步骤日志，避免长时间无诊断等待。

## 7. 风险与处理策略

| 风险点 | 影响 | 处理策略 |
|---|---|---|
| 三节点网络或代理配置异常 | Fabric、MySQL、后端跨节点访问失败 | 阶段 6 先验收 IP、hosts、SSH、Docker、代理绕过。 |
| MySQL 三节点权限不一致 | 后端写入失败或完整性误判 | 阶段 7 固定账号权限、字符集、时区，并做失败诊断测试。 |
| Fabric Gateway 证书或 TLS 配置错误 | 后端无法提交链码交易 | 阶段 8 先用 CLI 验证链码，再接后端 Gateway；阶段 9 确认证书路径需指向复制到 `node1` 的 gateway 材料。 |
| Fabric Gateway/gRPC/protobuf 版本不匹配 | 后端启动时报 `RuntimeVersion$RuntimeDomain` 类缺失 | 显式固定 Fabric Gateway 运行时依赖并检查 `mvn dependency:tree`。 |
| MyBatis 多数据源 mapper 未注册 | `node2/node3` 写入时报 MapperRegistry 异常 | 每个 `SqlSessionFactory` 显式注册 mapper 包，阶段 9.5 继续审核同类配置。 |
| 注解 SQL 转义混用 | 时间范围查询在 MySQL 报语法错误或动态 SQL 解析失败 | 区分普通注解 SQL 与 `<script>` 动态 SQL：普通 SQL 使用真实运算符，动态 SQL 中保留 XML 实体。 |
| MySQL JSON 格式化改变 normalized_message 文本 | 封存或完整性校验误报 leaf hash 不一致 | 取回 JSON 后反序列化为 CanonicalLog，再用统一 canonical 序列化计算 leaf hash。 |
| Fabric 链码安装阶段依赖下载失败 | `peer lifecycle chaincode install` 长时间等待后失败，可能被包装成 Docker daemon 错误 | 优先查看 peer 日志；链码包应包含 `vendor/`，避免 Fabric 构建容器访问 `proxy.golang.org`。 |
| 三库写入不一致 | 后续比对误判篡改 | 后端统一生成 `log_id`、`batch_id`、`leaf_hash`，数据库禁止自行生成关键业务 ID。 |
| Merkle Root 不可复现 | 链上 Root 与本地重算不一致 | 固定字段顺序、编码、时间格式、排序规则、空值规则，并保留测试向量。 |
| Filebeat 日志采集延迟或路径错误 | 演示时无日志进入后端 | Tomcat 日志目录固定挂载，AccessLogValve 建议关闭缓冲。 |
| MySQL 毁痕存储过程误用 | 正常数据被破坏 | 存储过程只部署 node1 演示库，后端和前端均不提供篡改入口。 |
| 背书策略为 OR | 可信叙事弱于多组织共同背书 | 沿用既有最小可运行网络，论文中说明该策略边界。 |
| 阶段 9.5 seal 门禁漏检普通明文字段篡改 | 攻击者只改 `request_uri/raw_message` 但不改 leaf 时，旧门禁可能放过 | 已处理：引入统一副本记录验证哈希，覆盖稳定业务字段和明文字段；seal 前三库 digest 必须一致。 |
| 阶段 9.5 完整性差异分类不精确 | `abnormal_nodes` 或 `EXTRA_LOG/MODIFIED_LOG` 误判会影响答辩可信度 | 已处理：seal 后以 `node2/node3` 为可信参考副本，确保仅真实异常副本进入异常节点列表。 |
| 异步同步任务卡在 `IN_PROGRESS` | node2/node3 离线时 seal 可能长时间等待 | 降低数据库连接超时，增加 stale `IN_PROGRESS` 重置，并在手册中提供手动恢复 SQL。 |
