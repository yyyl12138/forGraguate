# Ledger State Model

## Purpose

This document freezes the ledger-facing batch evidence contract for the log traceability system.

The blockchain stores only batch-level evidence. It must not store plaintext logs, leaf hash lists, Merkle proofs, MySQL replica data, or frontend display-only fields. Plaintext logs remain in the three MySQL replicas on `node1`, `node2`, and `node3`.

## Fabric Context

The contract is designed for the existing Fabric baseline:

- Fabric Core: `2.5.4`
- Channel: `mychannel`
- Organizations: `Org1MSP`, `Org2MSP`
- Endorsement policy: `OR('Org1MSP.peer','Org2MSP.peer')`
- State database: CouchDB behind the peers on `node2` and `node3`

Stage 1 does not change the Fabric topology, channel, endorsement policy, or CouchDB deployment.

## State Key

Every batch evidence record is stored under a deterministic state key:

```text
batch:{batch_id}
```

Example:

```text
batch:bch_v1_tomcat-cve-2017-12615_20260422T020500Z
```

The chaincode must reject duplicate `batch_id` writes. A submitted batch must never overwrite an existing state value.

## Batch ID

`batch_id` is generated by the backend before writing MySQL replicas or ledger evidence:

```text
bch_v1_{source}_{yyyyMMddTHHmmssZ}
```

Rules:

- The timestamp segment is the UTC batch window start time.
- The timestamp segment uses second precision in the ID, even though stored time fields use millisecond precision.
- The first version prefix is `bch_v1`.
- `source` is the business log source, for example `tomcat-cve-2017-12615`.
- `source_node` is a MySQL replica identifier and must not be included in `batch_id`.

Example:

```text
bch_v1_tomcat-cve-2017-12615_20260422T020500Z
```

## Batch Window

Batches use fixed 60 second windows.

Window semantics:

```text
[start_time, end_time)
```

Rules:

- A log belongs to a batch if `start_time <= log.timestamp < end_time`.
- Empty batches are not sealed and are not written to the ledger.
- `start_time` must be strictly earlier than `end_time`.
- For the current project stage, `end_time - start_time` must be exactly 60 seconds.

## Time Format

All ledger timestamps use UTC with millisecond precision:

```text
YYYY-MM-DDTHH:mm:ss.SSSZ
```

Examples:

```text
2026-04-22T02:05:00.000Z
2026-04-22T02:06:00.000Z
```

Local timezone strings such as `2026-04-22 10:05:00 +0800` must not enter ledger state or hash input.

## BatchEvidence JSON

`BatchEvidence` is the ledger value stored at `batch:{batch_id}`.

```json
{
  "batch_id": "bch_v1_tomcat-cve-2017-12615_20260422T020500Z",
  "merkle_root": "36adaf679ffc6df79290894fa7213f1a1855c29a177a898bd94e67bd86cd944f",
  "log_count": 3,
  "start_time": "2026-04-22T02:05:00.000Z",
  "end_time": "2026-04-22T02:06:00.000Z",
  "source": "tomcat-cve-2017-12615",
  "schema_version": 1,
  "hash_algorithm": "SHA-256",
  "canonicalization_version": "clog-v1",
  "created_at": "2026-04-22T02:06:01.000Z",
  "tx_id": "fabric_tx_id_assigned_by_chaincode_context"
}
```

## Field Contract

| Field | Type | Required | Description |
|---|---:|---:|---|
| `batch_id` | string | yes | Deterministic backend-generated batch ID. |
| `merkle_root` | string | yes | Batch Merkle Root, 64 lowercase hex characters. |
| `log_count` | integer | yes | Number of canonical logs in the batch. Must be greater than `0`. |
| `start_time` | string | yes | Inclusive UTC batch start time, millisecond precision. |
| `end_time` | string | yes | Exclusive UTC batch end time, millisecond precision. |
| `source` | string | yes | Business log source, for example `tomcat-cve-2017-12615`. |
| `schema_version` | integer | yes | Ledger state schema version. Current value: `1`. |
| `hash_algorithm` | string | yes | Hash algorithm. Current value: `SHA-256`. |
| `canonicalization_version` | string | yes | Canonical log and Merkle contract version. Current value: `clog-v1`. |
| `created_at` | string | yes | UTC timestamp when the evidence record is created on the ledger. |
| `tx_id` | string | yes | Fabric transaction ID from chaincode context. |

## Chaincode Write Validation

`CreateBatchEvidence` must enforce these rules before `PutState`:

- State key must be exactly `batch:{batch_id}`.
- `batch_id` must match `^bch_v1_[a-z0-9][a-z0-9-]*_[0-9]{8}T[0-9]{6}Z$`.
- `merkle_root` must match `^[a-f0-9]{64}$`.
- `log_count` must be greater than `0`.
- `start_time`, `end_time`, and `created_at` must match UTC millisecond format.
- `start_time` must be earlier than `end_time`.
- The batch window must be 60 seconds for `schema_version = 1`.
- `source` must be non-empty and must match the source segment inside `batch_id`.
- `schema_version` must equal `1`.
- `hash_algorithm` must equal `SHA-256`.
- `canonicalization_version` must equal `clog-v1`.
- If `GetState(batch:{batch_id})` returns an existing value, the chaincode must reject the write.
- `tx_id` must be populated from Fabric chaincode context, not accepted as a trusted client input.

## Query Fields

CouchDB rich queries may use these stable fields:

- `doc_type`, if chaincode chooses to store it internally as `"BatchEvidence"`
- `batch_id`
- `source`
- `start_time`
- `end_time`
- `schema_version`
- `hash_algorithm`
- `canonicalization_version`

If `doc_type` is added by chaincode, it is an internal query helper and must not change the public `BatchEvidence` fields required by backend and frontend contracts.

Recommended CouchDB selector examples:

```json
{
  "selector": {
    "source": "tomcat-cve-2017-12615",
    "start_time": { "$gte": "2026-04-22T02:00:00.000Z" },
    "end_time": { "$lte": "2026-04-22T03:00:00.000Z" }
  }
}
```

```json
{
  "selector": {
    "source": "tomcat-cve-2017-12615",
    "schema_version": 1
  }
}
```

## Evidence Boundary

The ledger proves that a batch-level digest and its metadata were committed. It does not prove the truth of plaintext log content by itself.

Integrity checking must:

1. Read `BatchEvidence` from Fabric by `batch_id`.
2. Read plaintext logs for the same `batch_id` from `node1`, `node2`, and `node3` MySQL replicas.
3. Recompute each replica Merkle Root using `docs/hash-batch-contract.md`.
4. Compare each replica Root with `BatchEvidence.merkle_root`.
5. Use cross-replica comparison to locate deleted, inserted, or modified plaintext logs.

## Non-Ledger Fields

These fields must not be written into `BatchEvidence`:

- Plaintext log records
- `raw_message`
- `normalized_message`
- `leaf_hash` list
- Merkle Proof paths
- `source_node`
- MySQL auto-increment IDs
- Filebeat runtime offsets after sealing
- Frontend-only display status
- Demo tamper flags

## Compatibility Rules

Any future incompatible change to canonicalization or hash input must create a new `canonicalization_version`.

Any future incompatible change to the ledger JSON shape must increment `schema_version`.

Existing records with `schema_version = 1` and `canonicalization_version = clog-v1` must remain verifiable with this document and `docs/hash-batch-contract.md`.
